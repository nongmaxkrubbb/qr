# สรุปภาพรวมโปรเจค: ระบบคิวโรงพยาบาล (QR Queue)

โปรเจคนี้เป็นระบบบริหารจัดการคิวสำหรับโรงพยาบาล ที่เปิดให้คนไข้กดรับคิว แล้วใช้สมาร์ทโฟนสแกน QR Code เพื่อติดตามสถานะแบบ Real-time โดยไม่ต้องนั่งรอหน้าห้องตรวจตลอดเวลา โดยมีการผสานการทำงานระหว่าง **Python (Flask Framework)** และ **Database (SQLite)** เพื่อควบคุมตรรกะและสถานะของระบบดังนี้:

---

## 🐍 บทบาทของ Python (Backend & Logic)
Python ทำหน้าที่เป็นสมองกลหลักของระบบ โดยใช้ **Flask Framework** ในการทำงานต่างๆ ดังนี้:

1. **การควบคุมเส้นทาง (Routing & Web Server):**
   * ควบคุมการแสดงผลของหน้าเว็บ (UI) เช่น หน้าแรก หน้าเจ้าหน้าที่ หน้าแดชบอร์ด
   * แยกส่วนการทำงาน (Blueprints) เป็น `user_routes` สำหรับคนไข้ และ `admin_routes` สำหรับเจ้าหน้าที่เพื่อความปลอดภัยและง่ายต่อการจัดการ
2. **การสร้าง QR Code แบบ Dynamic (ไลบรารี `qrcode`):**
   * เมื่อคนไข้กดรับคิว Python จะดึงรหัสตั๋วไปสร้างเป็นภาพ QR Code 
   * โดยแปลงภาพ QR เป็น Base64 String ทำให้ส่งไปแสดงผลบนหน้าเว็บ (HTML) ได้ทันทีโดยไม่ต้องบันทึกเป็นไฟล์รูปลงในเครื่อง
3. **การคำนวณและประมวลผล (Business Logic):**
   * **คำนวณเวลารอโดยประมาณ (ETA):** ค้นหาข้อมูลใน Database เพื่อดูว่า 20 คิวย้อนหลังของแผนกนี้ ใช้เวลาตรวจเฉลี่ยกี่นาที แล้วเอาไปคูณกับจำนวนคิวที่กำลังรออยู่
   * **กำหนดหมายเลขคิวอัตโนมัติ:** เมื่อมีคนกดรับคิว Python จะเช็คว่าวันนี้แผนกนี้ (เช่น ตรวจโรคทั่วไป Prefix: B) ออกบัตรไปกี่ใบแล้ว และจัดการรันเลขใหม่เป็น `B001`, `B002` เสมอในแต่ละวัน
4. **ความปลอดภัย (Security):**
   * จัดการระบบ Login ของเจ้าหน้าที่ด้วย **Session** 
   * ทำการเข้ารหัสและตรวจสอบรหัสผ่านของแอดมินด้วยเทคนิค Password Hashing (`werkzeug.security`)
5. **การเป็น API ให้ฝั่งหน้าบ้าน:**
   * หน้าเว็บคนไข้จะยิงมาร้องขออัปเดตข้อมูลทุกๆ 5 วินาที Python จะดึงสถานะใหม่ล่าสุดจาก Database แล้วตอบกลับไปเป็น JSON

---

## 💾 บทบาทของ Database (SQLite)
ระบบใช้ **SQLite** เป็นฐานข้อมูล ทำหน้าที่จดจำ "สถานะ (State)" ของทุกสิ่งทุกอย่างที่เกิดขึ้น โดยแบ่งออกเป็น 3 ตารางหลัก:

1. **ตาราง `queue_types` (ข้อมูลแผนก)**
   * เก็บรายชื่อแผนกและตัวอักษรนำหน้า (Prefix) เช่น A สำหรับซักประวัติ, B สำหรับตรวจโรค
   * **ทำไมถึงต้องมี?** เพื่อให้เพิ่ม/ลดแผนกใหม่ในอนาคตได้ง่ายโดยไม่ต้องแก้โค้ด Python
2. **ตาราง `staff` (ข้อมูลบัญชีเจ้าหน้าที่)**
   * เก็บ Username และ Password ที่ผ่านการ Hash แล้ว
3. **ตาราง `tickets` (หัวใจหลักของระบบ)**
   * บันทึกบัตรคิวทุกใบที่ถูกสร้างขึ้น โดยเก็บข้อมูลสำคัญคือ:
     * `queue_number`: หมายเลขคิว (เช่น B005)
     * `queue_type_id`: เป็นคิวของห้องไหน
     * `status`: มีแค่ 4 สถานะหมุนเวียนกัน คือ `waiting` (รอเรียก) ➔ `in_service` (กำลังตรวจ) ➔ `done` (เสร็จ) หรือ `skipped` (ข้าม)
     * `created_at`, `called_at`, `completed_at`: ระบบ Time-stamp เพื่อจดจำเวลาเอาไว้ใช้คำนวณสถิติต่างๆ

---

## 🔄 ตัวอย่างวงจรการทำงานร่วมกัน
1. **คนไข้กด "ตรวจโรคทั่วไป"**: Python สั่ง Database ให้สร้าง `ticket` ใหม่พร้อมบันทึกเวลา `created_at`
2. **หน้าจอคนไข้**: Python ดึง ID ออกมาสร้างเป็นลิงก์ QR Code แล้วส่งให้คนไข้สแกน
3. **พยาบาลกด "เรียกคิว"**: Python สั่ง Database ค้นหาคิวที่เก่าที่สุดที่สถานะเป็น `waiting` แล้วอัปเดตเปลี่ยนเป็น `in_service` พร้อมสแตมป์เวลา `called_at`
4. **การคำนวณเวลา**: Python ดึงข้อมูลจาก `completed_at` ลบด้วย `called_at` ตลอดทั้งวัน เพื่อนำมาแสดงในหน้า Dashboard สถิติให้ผู้บริหารดู

การออกแบบแยกส่วนระหว่าง Logic และ State แบบนี้ ทำให้ระบบรองรับการใช้งานหลายอุปกรณ์ได้ (พยาบาลกดเรียกในแท็บเล็ต คนไข้ดูในมือถือตัวเอง) ข้อมูลก็ยังตรงกันแบบเรียลไทม์ครับ
